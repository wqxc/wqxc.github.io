开启事物处理。

所谓额事物：类似于 axios 的 请求拦截 加 网络请求 加 响应拦截

一次网络请求，先走 了 请求拦截 而后是 真正的 网络请求，再最后 处理一下响应信息。

这里的事物是一样的逻辑。

我们在 callback 的时候，其实就是 handleClick 的是，会先执行 一些前置处理。

而后才是 执行 callback。 callback 执行完毕了就是一些收尾处理。

这里的收尾处理就是 更新组件。

到了这一步，还只是 React 的一些内部的处理信息，接下来是调用 setState 的情况。

4、调用 setState

调用 setState 的时候，其实是调用了 ReactUpdateQueue.enqueueSetState

顾名思义，就是讲 state enqueue

5、enqueueSetState

拿到当前的组件实例，为实例上的 _pendingStateQueue 添加值。

_pendingStateQueue 是一些等待变更的 state 的队列。

这里，我们 setState 的内容，最终被存储在了 _pendingStateQueue 队列里。

而后 调用了 enqueueUpdate

6、enqueueUpdate

enqueueUpdate 函数的作用很简单，就是调用 ReactUpdates.enqueueUpdate(internalInstance);
参数就是当前的组件实例

7、ReactUpdates.enqueueUpdate

在第三步的时候 isBatchingUpdates 已经被修改为 true 了。这个时候，enqueueUpdate 函数的主要作用其实就剩下了一项：

dirtyComponents.push(component);

dirtyComponents 是一个数组，数组内存储的就是需要更新的组件。

8、多个 setState

如上 4-7 是调用 setState 的一个过程，就像是例子上我们调用了三次 setState。

那么 4-7 步就会循环执行

最后，handleClick 执行完毕了。

flushBatchedUpdates 函数的作用也很简单 就是 遍历 dirtyComponents，这里又开启了一个事物 ReactUpdatesFlushTransaction。

16、updateComponent – 判断是否需要更新

如果 _pendingForceUpdate (强制更新) 或者 是组件自己没有 定义 shouldComponentUpdate 方法 或者是 组件定义了方法

17、updateComponent – 更新
核心是调用了 _performComponentUpdate 方法

17-1、 _performComponentUpdate 如下

拿到之前的 state，之前的 props，而后看看组件是否有 componentWillUpdate 生命函数。

而后才是真正的更新组件方法 _updateRenderedComponent

最后 组件更新完了 就执行生命函数 componentDidUpdate

17-2、_updateRenderedComponent
这个方法主要做的就是 执行组件的 render 方法，而后相应的更新 DOM

根据 shouldUpdateReactComponent 方的返回结果，这个函数会做不同的事情。

shouldUpdateReactComponent 的作用是判断当前的组件是否需要更新。

17-3、_renderValidatedComponent

这个方法的作用是 调用组件的 render 方法的并返回一个结果，这个结果是一个 Element。
Element 可以是 对象，字符串 数字 或者干脆是 null

17-3、 shouldUpdateReactComponent

比对 preElement 和 nextElement 来决定组件是否需要更新，这个 shouldUpdateReactComponent 和组件的生命方法 shouldComponentUpdate 不同。

prevElement 和 nextElement 都不为 null 的时候，多做一些操作，否则就直接返回 false，表示 不需要更新

在 17-2 中，我们知道 这个方法返回 false 主要做的事情是

1、unmountComponent 卸载 之前的组件 prevComponentInstance

2、根据当前的 Element 等参数 实例化组件 _instantiateReactComponent

3、调用组件的 mount 方法生成 markup 也就是 html 文件

4、直接替换 prevComponentID 位置的 html 为 新的 html。

这个时候主要是因为，shouldUpdateReactComponent 返回了 false，表示两个组件完全不一样，React 认为两个组件类型都不一样了，就不考虑遍历对比复用内部组件，而是暴力的直接进行替换。

举个例子 ：
prevElement 为 null nextElement 是一个组件实例，这个时候应该无需进一步对比两者的区别，找出变动的部分更新，而是让 nextElement 直接替换 原来位置的 null 就好了。

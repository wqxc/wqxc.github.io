Redux 的进阶

优化 Redux

缺陷

Redux 中 Store 是 唯一的 ，而且，store 承担了很多的内容。

状态树

dispatch 一个 action

合并所有的 Reducer

所以我们要在 View 中改动 store 管理的状态的时候，都需要引入 Store。

如果我们在多个地方都需要的话，那么各个组件就都要引入 store，而且这个引入是相对路径。随着代码的层级越来越深，耦合度也就越来越深。store 文件的位置就不能轻易的挪动。

这种情况定然是有问题的。

解决办法 - context

因为 Store 需要多处引用的弊端，所以要解决这个弊端的方式是–在一个地方引入。比如：根组件。

可能的尝试：使用 props 来逐层传递 store，这个解决方案会导致那些不需要使用 store 的组件也得接收 store 并往下层传递。

我们可以使用 React 提出了 Context 功能 来解决这个问题。

Context 并不是 Redux 提供的解决方案，而是 借用 React 的 功能来解决 Redux 存在的缺陷。Context 顾名思义就是上下文环境的意思。它的作用是让树状的组件上所有的组件都能访问一个共同的对象。

为了实现所有的组件都能访问一个共同的对象，就需要上下级组件相互配合。具体的方式是

上级组件：
要宣称自己支持 context ，并且提供一个函数来返回代表 context 的对象

所有的子孙组件：

上级组件的下的所有子孙组件，只要宣称自己需要这个对象，就可以通过this.context 访问到这个共同的环境对    象。

使用 context 优化 Redux

根据原理，我们要在跟组件引入 store，为此，我们最好是自己建造一个组件来作为根组件，以此不影响其他的 React 组件。我们就让这个组件叫做 Provider context 的 提供者。

注意

1、需要注意的是 Provider 声明的 store 的类型

与子孙组件要用的 store 类型必须一致 否则就无法访问到 context

2、 如果你的状态是跨组件传递的，而且，并非是所有的组件都需要使用这个状态，你就可以考虑使用 context，因为 Context 像是一个全局的对象，如果一个地方改变了，就会影响别的地方。这个操作很危险。

Redux 之所以可以配合 context 来使用，是因为 Redux 的 store 是全局唯一的，而且，store 的改动也是极其严格的，所以使用 context 是很合适的。

但是，你不能滥用 context。

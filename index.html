生命周期函数 -- componentdidmount

componentdidmount

1、调用的时机是: render之后，组件被渲染到DOM中之后运行

2、一般会在这个生命函数里进行ajax请求

3、在这里添加订阅，在销毁的时候取消订阅

4、可以在这里直接调用setState，触发额外的渲染，但此渲染会发生在浏览器更新屏幕之前，
保证，即使有两次render，用户不会看到中间状态

5、不能依赖第四项，如果渲染依赖DOM节点的大或者是位置，可以使用
比如 modals tooltips 需要先判定要展示的位置，而后才展示

生命周期函数 -- componentWillMount

componentWillMount

1、该生命函数中执行this.setState 是不会触发第二次渲染的。

2、它的作用和constructor的作用其实是一样的。

3、React不推荐在 componentWillMount 中发送异步请求

4、componentWillMount 是在render之前执行的，如果在该函数中执行异步请求的话，不能保证在render之前完成请求（如果render之后完成了ajax请求，即使调用了setState，也不会再触发重新渲染了。请求的数据就不能被渲染在页面上。）

5、它的作用是在服务端渲染的时候，componentDidMount 不会被执行，因此可以在 componentWillMount 中执行ajax请求

6、不推荐在 componentWillMount 里ajax请求的另一个原因与同构有关，如果在该函数里ajax则会在服务和客户端各自调用一次

JS/ES6的继承

ES6继承

一句话，主要是做了两件事

class A extends B{}
A.proto === B; //继承属性
A.prototype.proto == B.prototype;//继承方法

es6 继承必须在 construcor 中调用super方法，否则就会报错，
因为子类是没有自己的this对象的，而是继承自父类的this，然后对其进行加工，不调用super的话，就得不到this对象，
因此，调用之后才可以使用this

默认情况下，子类会自动的调用 constructor 方法，并调用super

在继承的时候手动 写了 constructor 方法的话，会覆盖默认的 constructor 方法

React Immutability

背景

开发过程中会遇到，state为数组或者是对象的情况。
我们使用splice，push，pop，unshift等方法来修改数组，再调用setState来更新的时候，
如果shouldComponent 里做了是否更新的判断的话，组件永远也不会更新。
因为React 里的state 是 不可变的。

所以我们在用的时候需要用一些手段来保证，setState的数据是不可变的。

状态类型是不可变类型的

number，string，boolean，null,undefined

这些状态都是不可变的，使用的时候无需担心

状态类型是数组的时候

使用 push，pop，unshift，shift，splice等方法都会修改原数组，不能保证不可变。

但是，concat，slice，filter，或者 es6的扩展运算符，可以保证不可变性

状态类型为对象的时候

使用Object.assign 或者是 扩展运算符来操作

一些概念

横切关注点

部分关注点「横切」程序代码中的数个模块，即在多个模块中都有出现，它们即被称作「横切关注点（Cross-cutting concerns, Horizontal concerns）」。

简单的来说，就是我们的业务逻辑分散在一个组件的不同生命周期中。

比如：在componentDidMount 里 注册监听函数
早 ComponentWilUnMount 里取消注册监听
类似这种横跨多个生命周期函数的逻辑。叫做横切关注点

最早的时候，使用mixins 或者是高阶组件来解决这个问题。
后来：
vue3中出现了setUp 来解决这个问题
React16之后出现了React-Hook 来解决这个问题

派生state

派生的意思是说，一个事物是从另外一个事物的发展中分化出的。
派生state是说，state不是我们在constructor中定义的state，而是使用props派生的。

比如：

state = { email: this.props.email };

state email 派生自 props。

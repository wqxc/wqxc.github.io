API -- React.createContext

Context

参考文章[https://zh-hans.reactjs.org/docs/context.html#gatsby-focus-wrapper]

Context 提供了一个无需为每层组件手动添加props，就能再组件树间进行数据传递的方法。
一般父组件向子组件传值，使用props的方式，但是，如果组件的层级是下边这样的。

Fourth组件需要使用First组件里的一个属性，但是，Second Third 是不需要这个属性的，但是，
他们依旧需要接收这个属性，并向下传递。

Fourth 组件需要的属性每增加或者是减少一个，都得在这个组件树里全部改动一番。

Context 就可以解决这个问题，但是，Context的应用场景是 不同层级的组件都需要访问一样的数据，比如Redux 做的那样，

如果只是想简单的避免属性的层层传递，还有一些其他的方式，参考[../技巧/组件组合.md]

useEffect

useEffect

useEffect 也叫做 副作用钩子函数。顾名思义，他的作用就是用来执行一些副作用操作。

副作用

什么算是副作用呢？比如一些IO操作，HTTP请求等。

副作用和纯函数是一个相对的概念，纯函数是说返回值依赖于他的参数，给定相同的参数，必然返回相等的返回值。
而如何函数内部做了和运算返回值没有关系的事情，比如修改了全局变量，修改了传入的参数，发送请求，console，手动修改DOM等都是副作用

类比类组件

useEffect Hook 可以看做是 componentDidMount，componentDidUpdate 和 componentWillUnmount 这三个函数的组合。

举个例子

useEffect执行时机

useEffect 会在DOM更新之后调用，也就是 组件挂载 和 更新阶段执行，也就是 componentDidMount，componentDidUpdate 两个类的生命周期方法中。

其实我们无需去考虑生命周期，而只需要记住，useEffect 会在DOM更新之后调用，而无需在意它是mount还是update

useEffect 清理函数

useEffect会返回一个函数，这个函数会在下一次 useEffect执行的时候执行。
举个例子

如上所示，下一次 useEffect 执行的时候，上一次的 cleanup会执行，一般这个都是用来清理一些东西。

但是，这里需要注意一点的是，useEffect 可以接受第二个参数，一个数组，表示依赖的状态。

如果是空数组，表示 useEffect 只会执行一次，这个时候，return 的清理函数就没有意义，因为清理函数是在下一次useEffect执行的之前之前，这里只会执行一次，就没有下一次了，清理函数自然就不会执行了。

为什么要使用useEffect

class组件中，我们经常将相关的逻辑分散在不同的生命函数中，而将，不同的逻辑放在同一个生命函数中。

比如我们要添加事件订阅，在mount中addEventListener 在 Unmount中移除 removeEventListener。

同样的，我们在 mount中也会做很多别的操作，比如 网络请求。

注意

useEffect 调度的effect不会阻塞浏览器更新屏幕，这让应用看起来更快，大多数情况下effect不需要同步执行，
如果需要同步执行的话，则可以使用 useLayoutEffect Hook。

举个例子吧：

没次点击 div的时候，div内的值，会闪烁一下，从一个随机字符串到0，而后会再次变为一个随机字符串。

因为这里有一个 重置为0的过程，页面先会渲染出一个 0 而后又迅速的渲染出一个 随机数。由于更新很快，所以会闪烁。

使用 useLayoutEffect 则没有这个情况。

useLayoutEffect 是内部的代码执行完毕，才会去更新视图。

如果机体的性能比较好，则不会有闪烁的情况，但是你可以使用延迟的方式来展现

useLayoutEffect

函数签名与 useEffect相同，但是它会在所有的DOM变更之后同步调用effect。可以使用
它来读取DOM布局，并同步触发重渲染。

这个hook可能阻塞UI的更新，所以，官方是不建议使用的，一般会在useEffect中使用的，实在没有办法的情况下，才考虑使用 useLayoutEffect。

渲染与帧

帧的概念

参考文章：[https://musicfe.dev/react-hooks/]

我们可以认为每一次组件的渲染被认为是一帧

每一帧都拥有自己的状态

当然，这个状态在函数组件中表现的更为明显，举个例子：

我们定义了一个组件 Example2,考虑一下先点击一下 Delay setCount 按钮，而后在三秒内连续点击三次setCount按钮，最终 count 的值会是多少呢？
直观上来看count应该是 4。但是，事实上 count应该是1。

步骤如下：

1、点击 Delay setCount ，setTimeout函数等待执行
2、点击了三次setCount (当然，这是在seTimeout等待执行的三秒内点击的三次)，count变为3
3、seTimeout 定义的回调函数执行了，此时的核心是 回调函数里的count 是多少？

考虑一下，我们在点击 Delay setCount 按钮的时候，count是多少呢？ 此时是 0，应该是毫无疑问的。
也就是 三秒之后 执行 setCount(0 + 1);

接下来是点击了三次 setCount 按钮之后，count 变成了3

此时的count并不会影响setTimeout函数内的count值，这就是React中帧的概念，每一帧的状态都是没有一点关系的独立个体。

详细的分析一下

Delay setCount 点击的时候做了什么？
首先是 handleClick 被调用了 ，count是多少呢？我们可以认为是下边这样的。

接下来，第一次点击 setCount 按钮, count 被更新了 函数重新执行，以渲染。
count 变成了

接下来是第二次，第三次，函数依旧会重新执行，count 就变成了 3

最后，setTimeout的回调函数到期开始执行，执行的方法当然是

count 又恢复到了 1

类组件中的表现

函数组件 或者是叫做 hook组件，我们应该将其和类组件完全的分割开来，hook本身不是用来代替类组件的。同样的例子，我们看一下在类组件中的表现

调用一次Alert Count 按钮，而后点击 三次 setCount ，最后alert 的 count值是 4.

总结

为什么出现上边这样的差距呢？核心是 在于 this。

类组件有自己的实例，每一次的修改是在实例上修改的，this永远指向的是最新的实例，最新实例拥有最新的state。

函数组件没有自己的实例，每次渲染都是执行一次函数，函数执行完毕之后就被销毁了，每一次渲染，
虽然函数组件内的函数，属性都看着和原来的一样，但是，实际上他们已经被销毁之后又再次创建了。

当然，函数组件可以使用 useRef来实现跨组件的保存状态

API -- createRef

createRef

创建一个通过ref属性附加到React元素的ref

只能使用在 HTML元素 或者是 类组件中，函数组件中无效

背景

DOM时代，使用document.getElementById 来获取DOM，进而操作

但是在虚拟DOM时代，这样的方式是行不通的，所以，虚拟DOM都会提供一个ref属性来绑定真实DOM

使用

像下边这样拿到DOM

ref的值

1、ref用在html元素上的时候，current属性接收底层的DOM作为其值

2、ref用在类组件上的话，current 的值是 类组件的实例,因为拿到的是类组件的实例，所以可以调用实例里的方法，
如此便可以在父组件上操控子组件的方法

3、函数组件没有实例，所以也不会有ref

ref属性更新的时机

在componentDidMount之前，或者componentDidUpdate 触发之前更新ref属性的值

ref回调函数

React 将在组件挂载时，会调用 ref 回调函数并传入 DOM 元素，当卸载时调用它并传入 null。在 componentDidMount 或 componentDidUpdate 触发前，React 会保证 refs 一定是最新的

ref 属性接收一个回调函数，来讲DOM元素存储在组件的变量中

回调形式的ref，可以在组件之间传递

举个例子

在上面的例子中，Parent 把它的 refs 回调函数当作 inputRef props 传递给了 CustomTextInput，而且 CustomTextInput 把相同的函数作为特殊的 ref 属性传递给了 。结果是，在 Parent 中的 this.inputElement 会被设置为与 CustomTextInput 中的 input 元素相对应的 DOM 节点。

ref转发

在父组件中为子组件添加ref可以访问子组件的实例，但是，这样就打破了组件的封装原则，耦合加深，
所以可以使用ref转发来做到这样的事情。

ref转发就是允许组件接收ref，并将其向下转发给子组件。

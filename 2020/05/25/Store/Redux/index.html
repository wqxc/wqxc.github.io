Flux思想的升华

Flux 思想的继承者—Redux

背景

Flux 的思想是值得肯定的，但是，Flux 框架本身确实是饱受争议，Redux 在 Flux 的基础上进行了改进，成为了最受欢迎的框架，而且，没有之一。

其他的实现者比如：Reflux Fluxible 等。

一般我们看到 Redux 就会想当然的认为他是 React 的附属品。但是。这里实际上是不是的。其中 Redux 的 首字母 R 本身和 React 的首字母 R 是没有任何关系的。

基本原则

唯一数据源

保持状态为只读

数据的改变只能通过纯函数完成

解释一下

唯一数据源

不像是 Flux 那样允许存在多个 Store，Redux 从一开始就说明最好的处理方式是 只存在一个 Store，所有要接受管理的状态都挂载这个 Store 上。 当然，Redux 不会阻止你创建多个 Store，只是，多个 Store 本身不会有任何的好处，还不如只使用一个 Store 更容易组织代码。

保持状态只读

要想修改 Store 的状态，就必须要派发一个 Action 来完成。这一点和 Flux 是一样的。这里的只读不是，完全不能修改，而是要改变一个状态不是去直接修改这个状态，而是返回一个新的状态给 Redux，由 Redux 来完成新状态的组装。这一点和第三个原则关系密切。

数据的改变只能通过纯函数完成

纯函数 就是 Reducer 。

据说 Redux 这个名字的 前三个字 Red 就是指的 Reducer。

而 Redux 就是 Reducer + Flux

Reducer 是计算机中很常用的一个概念，比较贴切的解释是 归纳。无论你有多少个值，经过 Reducer 之后 最后只会有一个结果。同样的值，无论调用多少次 Reducer 的结果是一样的。

第一个参数 State 是当前的状态，第二个参数 action 是接收到的 action 对象，而 Reducer 要做的就是 根据 当前的 state 和 action 对象来产生一个新的 state。

Reducer 是一个纯函数，也就是没有任何副作用的函数。结果完全由两个参数决定，同样的参数产生同样的结果。

我们可以看到 Reducer 只是负责计算，不负责存储状态

Action

Redux 的 Action 构造函数在是简单的 return 一个 Action 对象。不需要触发 dispatch 方法是因为，Redux 只有一个 Store，就无需创造一个 Dispatcher 来辅助 dispatch 了。Redux 里 dispatch 是存在于 Store 上，直接在 Store 上进行 dispatch 就好了。

Redux 在很多的地方都是如此，不做多余的处理，而是简简单单的返回一个对象。

总结一下

与 Flux 相比之下

ActionType 是一样的

Action 是不同的

Flux的Action 会调用Dispatcher的dispatch方法分发 action对象


Redux 则只会返回一个普通的action对象

Store 是不同的

Flux可以有多个Store，而且需要Dispatcher类来辅助

Redux 则是只有一个Store，无需其他的辅助

分发 action 对象的方式不同

Flux 需要使用辅助类 Dispatcher来辅助分发

Redux则是完全在Store上进行分发

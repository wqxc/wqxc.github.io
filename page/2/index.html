不可变值

setState 必须使用不可变值：

举个例子吧：

this.state.list.push(‘d’) 就不是 不可变量。

push会修改原数组，当 this.state.list.push(‘d’) 的时候state.list 就已经被修改了

如果使用了变量会发生什么。

一般而言，即使在setState里使用了变量也没有影响，但是，在性能优化的时候就会有问题。

也就是使用shouldComponent函数的时候

shouldComponentUpdate 中可以决定是否渲染组件，我们这里比较了 nextState.list 和 this.state.list

因为使用了可变量 this.state.list ==[‘a’, ‘b’, ‘c’,’d’]

nextState.list 也等于 [‘a’, ‘b’, ‘c’,’d’] 所以 shouldComponentUpdate 永远都返回false，组件不会渲染

解释一下：nextState.list 和 this.state.list 为什么都等于 [‘a’, ‘b’, ‘c’,’d’]

首先：nextState.list == [‘a’, ‘b’, ‘c’,’d’] 是因为

而this.state.list == [‘a’, ‘b’, ‘c’,’d’]

是因为

所以这里要使用不可变数据，
比如：

this.setState({
list: this.state.list.concat([‘d’])
})

concat 方法不会修改原数组。

这里的不可变两更多的是，对于引用数据的处理。

注意

在使用引用类型的数据的时候，一定要注意，修改的方式。

生命周期函数 -- shouldComponentUpdate

shouldComponentUpdate 具有拦截渲染的租用

shouldComponentUpdate 接收两个参数：

nextProps，nextState

nextProps:接下来要渲染的组件的props(父组件传来的,本次渲染是因为父组件传递的props发生变化而引起的)
nextState: 接下来要渲染的组件的 state（state是组件自身）

一般我们会和 this.state来对比。

渲染结束之后 this.state才会发生变化，所以，this.state我们可以将它认为是当前的state状态，nextState是接下来state的状态，
通过对比来确认是否需要更新。

需要更新就返回 true
否则就返回false

1、https

vue3中开启https会导致热更新失效

控制台报错

VUE报错代码： sockjs.js:1606 GET https://192.168.1.102:8080/sockjs-node/info?t=1615612302391 net::ERR_CONNECTION_REFUSED

sockjs 是被用来和本地进行通信的，这个通信中断了，HMR就失效了，代码更改了之后，进行保存之后是无法热更新的，需要手动更新。

我这里出现这个的问题是因为在本地开启了https，因为要测试Facebook登录，必须要使用https

因为开启了https，但是没有证书所以通信是失败的。

我们需要做的不是去node_modules里将代码注销掉，而是手动的安装一个证书

安装一个生产证书的工具 mkcert 环境是 Mac

brew install mkcert

进入项目中（将证书生成存在项目的根目录下）

mkcert -install

配置一下vue.config.js

devServer: {
proxy: process.env.VUE_APP_PROXY,
https: {
key: fs.readFileSync(path.resolve(dirname, ‘./key.pem’)),
cert: fs.readFileSync(path.resolve(dirname, ‘./cert.pem’))
}
}

mkcert local.shanyue.tech 命令可以查询生成的证书信息

mkcert 生成的证书名字是
cer ====> ./local.shanyue.tech.pem
key ====> ./local.shanyue.tech-key.pem
这里生成的证书名字我是修改cheng key.pem 和 cert.pem

如果配置之后发现控制台依旧在报错，查看一下 key 和 cert 对应的证书是否正确

如果是正确的，依旧报错，重启一下电脑，重新跑一次项目，应该就OK了

React 更新逻辑

setState

关键点:

setState不会立刻改变React组件中state的值；
setState通过引发一次组件的更新过程来引发重新绘制；
多次setState函数调用产生的效果会合并。

更新

不像是是vue监听响应式属性的变化来引发watcher，进而重新渲染。
React 则是依靠setState 这个API来引发一次组件的更新过程，进而渲染页面

多次调用setState 会引发批处理
isBatchingUpdate 会被设定为true

setState 引发的生命周期函数

shouldComponentUpdate
componentWillUpdate
render
componentDidUpdate

+++++++
顺带一说，props变化引发渲染，所调用的的生命周期函数比state多了一个：

componentWillReceiveProps
+++++++

shouldComponentUpdate return false 的时候 state就已经发生了变化

shouldComponentUpdate return true 的时候 state还没有变化
componentWillUpdate state也没有变化
render 这个阶段 state才更新了

所以说，我们setState的时候，是没法立刻拿到更新后的state的值得

setState 缺点

唯一的缺点就是 不像是字面上理解的那样理解修改state，而是延迟处理。

但是，依旧有方法解决这个问题。

那就是接受一个函数来作为参数

setState({count: count + 1})
setState({count: count + 1})
setState({count: count + 1})

同一时间调用三次，最终也只会执行一次 count +1

想要达到加三次的效果，就需要使用函数参数

function increment(state,props){
retun {count: state.count + 1}
}

setState(increment)

setState(increment)

就可以实现加两次。

另外的一种方式是使用 addeventListener 添加事件，

addeventListener 添加的事件，不会开启批处理，也就是不会合并state

React

React 更新逻辑

更新逻辑

React的组件分两种，类组件和函数组件。
类组件中state变动之后，就会执行render方法，
函数组件则会重新将函数执行一遍。

函数组件里的一些问题：
函数组件里 在 hook里，使用 addEventListener 注册的点击事件，如果不清理的话，
则渲染一次就会绑定一个，渲染三次就绑定三个方法，点击一次，也会执行三次。

因为函数执行，函数内的代码，也会被执行，执行前后的代码是一样的，但是在React看来则是不同的。

总结一下

更新的时候：
类组件，只会执行组件内的render方法，渲染DOM

函数组件，则会执行函数本身。

模块化

概念

所谓的模块化，就是拆分和归类，将原本成百上千行代码放在一个文件拆分后重新组合为一个个独立的模块。

一个模块就是实现了某种功能的代码，有了模块化之后就能更方便的使用别人的工具。也叫做–轮子

模块化的核心是 低耦合高内聚。

模块化的规范

commonjs规范

在CommonJs规范中：

一个文件就是一个模块，拥有单独的作用域；

普通方式定义的变量、函数、对象都属于该模块内；

通过require来加载模块；

通过exports和modul.exports来暴露模块中的内容；

所有代码都运行在模块作用域，不会污染全局作用域；
模块可以多次加载，但只会在第一次加载的时候运行一次，然后运行结果就被缓存了，以后再加载，就直接读取缓存结果；
模块的加载顺序，按照代码的出现顺序是同步加载的;

浏览器不兼容CommonJS的根本原因，在于缺少四个Node.js环境的变量。

module
exports
require
global

简单的来说，commonjs是同步的模块加载，更加适用于服务端，因为全部的资源都是在服务端的硬盘中存储的，
但是，浏览器端却是不行，因为全部的资源都需要从服务器获取，受制于网速，同步加载会导致浏览器假死。

核心：CommonJS是同步的，只有加载完毕才会执行后边的操作，这一点在服务器端是无所谓的，因为所有的资源都是在本地存储的，加载资源的时间几乎可以忽略不计。

而浏览器端却是不行，以为浏览器加载资源需要从服务器下载，受制于网络状况，如果主线程一直在下载资源，就会导致页面假死。

所以，浏览器端有了自己的规范比如 AMD。

module对象

1、module.exports属性

表示当前模块对外输出的接口其他的文件加载该模块，
实际上就是读取这个属性的值。

2、exports 变量

node为每一个模块都提供了一个 exports属性，
他指向module.exports 相当于在模块中有一句话：
var exports = module.exports

在使用的时候可以为 exports 添加 属性比如：exports.add = xxx;

你不能这样写：exports= xxxx

这样就相当于切断了 exports = module.exports

有点类似于 js里的 prototype属性。

func.prototype.add = xxx

func.prototype = {}

一个是 为 prototype添加属性，一个是 直接将 prototype给换了。

最后，要导出，一定还是要用 module.exports而不能使用 exports。

标准的模块化

以上的这些只是前端社区自己的实现，并非是标准，而es6则是提出了前端模块化的标准。也就是 export 和 import

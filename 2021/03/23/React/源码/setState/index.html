3、batchingStrategy

如上所示，如果当前已经在批处理了，就执行执行方法。也就是我们的handleClick，
而，没有批处理，也就是我们在项目初始化后，页面渲染完成了,初次点击了button按钮。
这个时候:

开启事物处理。

所谓额事物：类似于axios的 请求拦截 加 网络请求 加 响应拦截

一次网络请求，先走 了 请求拦截 而后是 真正的 网络请求，再最后 处理一下响应信息。

这里的事物是一样的逻辑。

我们在 callback的时候，其实就是 handleClick的是，会先执行 一些前置处理。

而后才是 执行 callback。 callback 执行完毕了就是一些收尾处理。

这里的收尾处理就是 更新组件。

到了这一步，还只是React的一些内部的处理信息，接下来是调用setState的情况。

4、调用setState

调用setState的时候，其实是调用了 ReactUpdateQueue.enqueueSetState

顾名思义，就是讲state enqueue

5、enqueueSetState

拿到当前的组件实例，为实例上的 _pendingStateQueue 添加值。

_pendingStateQueue 是一些等待变更的state的队列。

这里，我们setState的内容，最终被存储在了 _pendingStateQueue 队列里。

而后 调用了 enqueueUpdate

6、enqueueUpdate

enqueueUpdate 函数的作用很简单，就是调用 ReactUpdates.enqueueUpdate(internalInstance);
参数就是当前的组件实例

7、ReactUpdates.enqueueUpdate

function enqueueUpdate(component) {
if (!batchingStrategy.isBatchingUpdates) {
batchingStrategy.batchedUpdates(enqueueUpdate, component);
return;
}

dirtyComponents.push(component);
}

在第三步的时候 isBatchingUpdates 已经被修改为true了。这个时候，enqueueUpdate 函数的主要作用其实就剩下了一项：

dirtyComponents.push(component);

dirtyComponents是一个数组，数组内存储的就是需要更新的组件。

8、多个setState

如上 4-7 是调用setState的一个过程，就像是例子上我们调用了三次 setState。

那么 4-7步就会循环执行

最后，handleClick执行完毕了。

9、尾部处理

回到第三步，React内部开启了 事物处理机制：transaction.perform

张下边这样：

可以认为 initialize 是axios里的 请求拦截器里添加的一个拦截函数
close 则可以被认为是 响应拦截器里添加了一个拦截。

如上是 批处理的 事物处里，这里是加了两个。

抽离一下：
一个事物的机制就如下这样：

[emptyFunction, emptyFunction, handleClick, ReactDefaultBatchingStrategy.isBatchingUpdates = false, ReactUpdates.flushBatchedUpdates.bind(ReactUpdates)]

React会一个个的执行，一个执行完执行下一个。

handleClick 执行完毕了，接着就是 将 isBatchingUpdates 字段修改为 false，表示，批处理结束了。

然后是重点：ReactUpdates.flushBatchedUpdates.bind(ReactUpdates)

10、flushBatchedUpdates

flushBatchedUpdates 函数的作用也很简单 就是 遍历 dirtyComponents，这里又开启了一个事物 ReactUpdatesFlushTransaction。

调用 runBatchedUpdates 来更新组件

11、runBatchedUpdates

核心是调用了 performUpdateIfNecessary 方法

12、performUpdateIfNecessary

核心 是调用了 internalInstance 组件实例的 performUpdateIfNecessary 方法。
这里的关键是确认 组件实例的方法 performUpdateIfNecessary 是什么时候添加上去的，有啥作用。

根据他的名称，我们完全可以猜出来他的作用是 看看组件是否需要更新，需要就更新，不需要就不更新。

组件在实例化的时候就会包含这些方法会被混入组件。

16、updateComponent – 判断是否需要更新

如果 _pendingForceUpdate (强制更新) 或者 是组件自己没有 定义 shouldComponentUpdate 方法 或者是 组件定义了方法

状态管理的背景

状态管理

背景

借鉴伟人的一句话：一切的前端高深概念都是纸老虎。

先说说背景吧。所谓的状态就是数据。

为什么会有状态管理这个东西呢？

我们先从简单的开始。

想象一下 一个组件 Parent 有俩子组件 Child1 和 Child2

俩子组件内都有一个计数器，每当用户点一下，计数器就加 1.

这个时候呢，我想要在 Parent 组件里计算出 俩计数器的和。

你咋办？

简单的来说，就是 每当用户点击的时候 将计数器的值传给 父组件 Parent 就好了。Vue 或者 React 等现代的框架都很容易做到。

这很简单。但是，如果 计数器存在的组件 不是 parent 的 组件怎么办？或者干脆是两个八竿子打不着的组件怎么办？

其实也有办法。我们定义一个全局的变量。

比如 建一个文件 叫做 global.js 在这个里边定义变量 count1 count2 对应两个计数器的值。

在两个计数器的组件引入这个文件。当计数器加 1，就将 count1 或者 count2 加 1

最后在 Parent 组件里也引入 global.js 将两个 count 加一块。

这也很简单。

但是，问题来了，计数器组件改动了数据，parent 怎么知道该不该做加法计算出来值。

因为是全局变量，谁都可以改动，那很容易出现问题。而且出现了错误也不容易查找。

所以才有了状态管理的框架。

其实简单的来说，这些状态管理的框架其基本的原理也只是一个全局变量而已，只是这个全局变量的访问更为严格，不是谁想访问就能访问的。即使访问也要按照规矩来一步一步的进行。

后来

Flux 框架 更像是一个规范，虽然他本身也是一个状态管理的库。但是，他的伟大之处在于，提出了状态管理应该遵循的规范。

而 Redux Vuex 等状态管理框架只是遵循 Flux 的基本规范的前提下进行的升华。

他们本质上是一样的。

追本溯源

我们知道所谓的 状态管理 这个牛掰轰轰的概念，其实也就是 全局变量管理。

无论是 Redux 还是 Vuex 都是遵循了 Flux 提出的理论。

而 Flux 提出的理论是为了解决 MVC 思想在前端应用的问题。

接下来我们再看看 MVC 和 Flux
